========================================
ESCENARIOS DE PRUEBA - Todo API
https://todo.pixegami.io
========================================

Fuente del contrato: https://todo.pixegami.io/openapi.json (OpenAPI 3.0.2, FastAPI)

Schema PutTaskRequest:
  - content   (string)  REQUERIDO
  - user_id   (string)  opcional
  - task_id   (string)  opcional
  - is_done   (boolean) opcional, default: false

Errores de validacion retornan: 422 Unprocessable Entity
Errores de recurso no encontrado retornan: 404 Not Found

--- ERROR CASES ---

Create Task (PUT /create-task):
[ ] Payload vacio {}                          - Esperado: 422 (content es requerido)
[ ] Falta campo content                       - Esperado: 422 (unico campo requerido segun schema)
[ ] content como string vacio ""              - Esperado: 200 o 422 (la API no documenta minimo de longitud; comportamiento a confirmar)
[ ] is_done como string en vez de bool       - Esperado: 422 (falla validacion de tipo)
[ ] is_done como integer (0 o 1)             - Esperado: 200 (FastAPI acepta 0/1 como bool) o 422 (a confirmar)

Get Task (GET /get-task/{task_id}):
[ ] task_id inexistente (UUID aleatorio)      - Esperado: 404
[ ] task_id con espacios o caracteres raros   - Esperado: 404 o 422 (depende del router de FastAPI)

Update Task (PUT /update-task):
[ ] Payload vacio {}                          - Esperado: 422 (content es requerido en PutTaskRequest)
[ ] Payload solo con content, sin task_id     - Esperado: 200 (task_id es opcional segun schema) — verificar comportamiento real de la API
[ ] task_id en payload que no existe          - Esperado: 404
[ ] content como string vacio ""              - Esperado: 200 o 422 (a confirmar, misma incertidumbre que en create)

Delete Task (DELETE /delete-task/{task_id}):
[ ] task_id inexistente                       - Esperado: 404
[ ] Doble delete (eliminar tarea ya eliminada)- Esperado: 404 en el segundo DELETE
[ ] task_id con espacios o caracteres raros   - Esperado: 404 o 422

List Tasks (GET /list-tasks/{user_id}):
[ ] user_id de usuario sin tareas             - Esperado: 200, "tasks": []
[ ] user_id con caracteres especiales         - Esperado: 200 o 422 (el router puede rechazarlo)


--- EDGE CASES ---

[ ] content extremadamente largo (10000+ chars) - PUT /create-task -> comportamiento indefinido; puede retornar 200 o 413 (API no documenta limite)
[ ] content con caracteres unicode y emojis     - PUT /create-task -> Esperado: 200, verificar que el contenido se preserva exacto en GET
[ ] content con solo espacios "   "             - PUT /create-task -> Esperado: 200 o 422 (a confirmar si la API valida blank-only strings)
[ ] Flujo completo CRUD en secuencia            - create -> get -> update -> get (verificar cambio) -> delete -> get (404)
[ ] Crear multiples tareas para el mismo user_id- PUT /create-task x3 con mismo user_id, luego GET /list-tasks/{user_id} -> "tasks" tiene 3 elementos


--- BOUNDARY CASES (PutTaskRequest) ---

Nota: el schema no define restricciones de longitud en ningún campo.
Los siguientes casos exploran los limites implicitos del tipo string:

[ ] content de 1 caracter                    - PUT /create-task -> Esperado: 200 (minimo semantico)
[ ] content de 255 caracteres exactos        - PUT /create-task -> Esperado: 200 (limite comun de VARCHAR)
[ ] content de 256 caracteres               - PUT /create-task -> Esperado: 200 (confirmar que no hay truncado ni rechazo)
[ ] user_id de 1 caracter                   - PUT /create-task con user_id de 1 char -> Esperado: 200
[ ] task_id con formato no-UUID en update   - PUT /update-task con task_id="abc123" (no UUID) -> Esperado: 404 o 200 (a confirmar)

  Path param: task_id string REQUERIDO
